# Java 后端学习笔记 04：MySQL、Redis 与缓存设计（详细版）

## 1. 学习目标
- 掌握 MySQL 建模、索引与慢查询优化方法。
- 掌握 Redis 数据结构、典型场景与风险控制。
- 能设计稳定的缓存策略，规避穿透、击穿、雪崩问题。
- 能将 MySQL 与 Redis 协同用于课程助教 Agent 项目。

## 2. MySQL 建模原则
### 2.1 表结构设计
- 主键：推荐 bigint 自增或雪花 ID（视分库分表策略）。
- 通用字段：`create_time`、`update_time`、`is_deleted`。
- 业务字段命名语义清晰，避免缩写歧义。

### 2.2 范式与反范式
- 范式有利于数据一致性。
- 反范式有利于查询性能。
- 实际工程按“读写比例、实时性、维护成本”折中。

### 2.3 软删除
- 使用 `is_deleted` 标记删除。
- 查询默认加条件过滤。

## 3. MySQL 索引体系
### 3.1 索引类型
- 主键索引
- 唯一索引
- 普通索引
- 联合索引

### 3.2 联合索引与最左前缀
- 索引 `(a,b,c)` 可命中 `a`、`a,b`、`a,b,c`。
- 跳过左列会导致索引失效。

### 3.3 索引失效常见场景
- 对索引列做函数操作。
- 隐式类型转换。
- 模糊查询 `%xx` 前置通配。

### 3.4 EXPLAIN 关键字段
- `type`：访问类型，越接近 `const` 越好。
- `rows`：扫描行数。
- `Extra`：是否 `Using index`、`Using filesort` 等。

## 4. 事务与隔离级别
- ACID：原子性、一致性、隔离性、持久性。
- 常见隔离级别：读未提交、读已提交、可重复读、串行化。
- InnoDB 默认可重复读。

常见问题：
- 脏读、不可重复读、幻读。
- 长事务导致锁等待与主从延迟。

## 5. Redis 核心知识
### 5.1 数据结构与场景
- String：缓存对象、分布式锁标记。
- Hash：对象字段缓存。
- List：队列。
- Set：标签去重。
- ZSet：排行榜、延时任务。

### 5.2 持久化机制
- RDB：快照，恢复快但可能丢失最近数据。
- AOF：追加日志，数据更完整。
- 可按业务选择混合策略。

### 5.3 过期策略与淘汰策略
- 过期策略决定 key 到期何时清理。
- 淘汰策略决定内存不足时淘汰谁（LRU/LFU/随机等）。

## 6. 缓存设计：Cache-Aside 模式
### 6.1 读流程
1. 先查 Redis。
2. 未命中则查 MySQL。
3. 回填 Redis 并设置 TTL。

### 6.2 写流程
1. 更新 MySQL。
2. 删除对应缓存 key。

注意：
- 建议“删缓存”而非“更新缓存”，降低双写复杂度。

## 7. 三大缓存问题与方案
### 7.1 缓存穿透
现象：请求大量不存在数据，缓存与数据库都被击穿。
方案：
- 空值缓存。
- 布隆过滤器。

### 7.2 缓存击穿
现象：热点 key 失效瞬间并发打库。
方案：
- 互斥锁重建缓存。
- 热点 key 永不过期 + 后台异步刷新。

### 7.3 缓存雪崩
现象：大量 key 同时过期。
方案：
- TTL 加随机值。
- 多级缓存与限流降级。

## 8. 课程助教 Agent 项目中的落地建议
- MySQL 存储：用户、会话元信息、反馈、问题分类标签。
- Redis 存储：对话短期记忆、热门问题答案缓存、限流计数。
- 向量库存储：知识片段 embedding（可与主库解耦）。

## 9. 监控与治理
- MySQL：慢 SQL 日志、连接池监控、QPS/RT 指标。
- Redis：命中率、内存、慢查询、连接数。
- 缓存命中率低时优先排查 key 设计与 TTL 方案。

## 10. 常见错误清单
- 把大对象直接塞 Redis，导致网络与序列化开销大。
- 使用不稳定 key 规则导致缓存命中率低。
- 在事务未提交前删除缓存，造成脏读窗口。
- 未设置超时导致缓存永久膨胀。

## 11. 实验与练习题
1. 设计课程详情缓存并实现缓存旁路策略。
2. 使用 `EXPLAIN` 优化一个慢查询。
3. 实现“每用户每分钟最多 5 次提问”的 Redis 限流。
4. 对比“无缓存”和“有缓存”接口响应时间。

## 12. 面试问答
- MySQL 联合索引如何命中？
- Redis 为什么快？
- 缓存一致性如何保障？
- 缓存击穿和雪崩的区别？

## 13. 本章总结
MySQL 保障数据正确性，Redis 提升访问性能。两者不是替代关系，而是协同关系。后端工程的关键在于：清晰的数据分层、合理的缓存策略、持续的监控优化。
