# Java 后端学习笔记 05：AI Agent、RAG、工具调用与 MCP（详细版）

## 1. 学习目标
- 理解 AI Agent 在后端系统中的角色与边界。
- 掌握 RAG 全流程：文档构建、切分、向量化、检索、生成。
- 理解 Tool Calling 与 MCP 服务的集成方式。
- 能设计一个可运行、可评估、可写进论文的课程助教 Agent。

## 2. Agent 系统分层架构
一个可落地的 Agent 后端建议分为 5 层：
1. 接入层：REST API / WebSocket。
2. 编排层：Prompt、Memory、Advisor、Tool Router。
3. 模型层：ChatModel / EmbeddingModel。
4. 数据层：MySQL、Redis、向量库。
5. 观测层：日志、指标、链路追踪、评测集。

## 3. RAG 全流程详解
### 3.1 文档准备
- 来源：课程讲义、实验手册、FAQ、教师答疑记录。
- 要求：结构化、术语统一、减少噪音文本。

### 3.2 文档切分（Chunking）
- 固定长度切分：实现简单，但易断语义。
- 语义切分：按标题/段落/主题切分，质量更高。
- 建议：chunk 300-800 汉字，带 overlap（如 50-120 字）。

### 3.3 向量化（Embedding）
- 用 embedding 模型把文本转为向量。
- 同语义文本在向量空间距离更近。

### 3.4 向量存储
- 可选：pgvector、Milvus、Elasticsearch 向量检索。
- 保存字段建议：`chunk_id`、`source`、`title`、`content`、`embedding`、`metadata`。

### 3.5 检索与重排
- 检索 topK 候选片段（如 top5/top10）。
- 可加重排模型（rerank）提升相关性。
- 拼接高相关上下文给大模型。

### 3.6 增强生成
- Prompt 中明确要求“仅基于检索内容回答”。
- 输出可附来源片段或出处，降低幻觉风险。

## 4. 对话记忆（Memory）设计
### 4.1 短期记忆
- 保存最近 N 轮对话，保持上下文连贯。
- 可用 Redis 存储，按会话 ID 分桶。

### 4.2 长期记忆
- 提炼用户偏好、能力画像、学习路径。
- 存 MySQL，支持跨会话复用。

### 4.3 记忆污染风险
- 错误信息反复进入上下文会放大偏差。
- 需要清洗策略：过期、总结、置信度标注。

## 5. Advisor 设计（以 Spring AI 为例）
Advisor 可理解为“请求/响应拦截与增强链”：
- 安全 Advisor：违禁词与风险指令拦截。
- Memory Advisor：自动注入会话上下文。
- Logging Advisor：记录模型请求、响应、耗时。
- RAG Advisor：自动检索并注入知识片段。

## 6. Tool Calling 设计
### 6.1 何时需要工具调用
当答案不在模型参数内或需实时数据时：
- 查询课程表
- 查询作业截止时间
- 查询实验室开放时段

### 6.2 设计原则
- 工具输入输出 schema 清晰。
- 参数要可校验（课程号、日期范围）。
- 设置超时、重试、熔断、兜底回复。

### 6.3 典型执行链
1. 用户提问。
2. 模型判断需调用工具。
3. 后端执行工具并返回结构化结果。
4. 模型基于工具结果组织自然语言回答。

## 7. MCP 服务集成思路
MCP（Model Context Protocol）适合把外部能力标准化：
- 把“查课表”“查成绩规则”“查实验资源”封装成 MCP 工具。
- Agent 通过统一协议发现与调用。
- 优点：降低耦合，便于扩展多个工具源。

## 8. 课程助教 Agent 功能蓝图
### 8.1 必做能力
- 多轮对话
- 基于本地知识库的问答（RAG）
- 风险问题拦截
- 对话记忆持久化

### 8.2 可选增强
- 学习路径推荐
- 错题诊断
- 自动生成练习题与答案解析
- 工具调用查询课程实时信息

## 9. 评估与实验设计（论文可直接用）
### 9.1 指标
- 命中率：检索结果与问题相关度
- 准确率：回答正确性（人工标注）
- 幻觉率：无依据陈述比例
- 时延：P50/P95 响应时间
- 成本：平均 token 消耗

### 9.2 对照实验
- 无 RAG vs 有 RAG
- 不同 chunk 大小对准确率影响
- 不同 topK 对时延与质量影响
- 是否启用重排模型

### 9.3 错误分析
- 检索不准：文档切分/索引策略问题
- 回答跑偏：system prompt 不稳定
- 工具失败：参数解析或超时处理不足

## 10. 安全与合规建议
- 禁止输出违法、危险内容。
- 对用户输入做敏感词和注入防护。
- 日志脱敏（key、token、隐私字段）。
- 对模型调用失败做降级提示。

## 11. 常见工程问题
- 把所有逻辑塞一个 service，难维护。
- 没有离线评测集，改动效果不可量化。
- 缺少会话隔离，导致用户上下文串话。

## 12. 实施路线（0-1）
1. 先做无 RAG 单轮聊天跑通。
2. 加 Memory 形成多轮对话。
3. 加本地文档切分与向量检索。
4. 加 Advisor（日志、安全、检索增强）。
5. 加 Tool Calling + MCP。
6. 建立评测集并做论文实验。

## 13. 面试问答

- topK 调大会发生什么问题？

- MCP 在工程上解决了什么痛点？

## 14. 本章总结
Agent 项目要想真正可用，核心不在“接上模型”，而在工程化编排能力：检索、记忆、工具、评估、安全缺一不可。你当前课程助教项目已经有基础框架，下一步重点是把 RAG 与工具调用做成可验证的闭环。
